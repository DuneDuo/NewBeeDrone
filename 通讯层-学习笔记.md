## 二、提问路线回顾

> 以下是你在学习通讯层过程中自然形成的提问路径。这条路径本身就是一套高效的学习方法论——**从全局到局部，从物理到协议，从理解到实操**。

### 第 1 问：「项目怎么分成不同层？」

```
Q: "我可以不可以把整个项目按分工分为飞机层，电站层，通讯层，服务层"
```

**这一问的价值**：建立了全局坐标系。在学习任何子系统之前，先搞清楚"我在哪，我的上下邻居是谁，我和他们用什么语言交流"。这是**结构化思维**的起点。

**学到的方法**：
- 拿到一个复杂系统，第一件事不是钻细节，而是**画分层框图**
- 每层定义清楚：职责边界 + 对外接口 + 依赖关系
- 分层之后，每层可以独立学习、独立测试、独立排错

### 第 2 问：「完整的通信链路是什么样的？」

```
Q: "给我飞机->树莓派->网卡->服务器的通信链路，以及mavlink是怎样实际给飞控发指令的"
```

**这一问的价值**：从分层走向端到端。分完层之后马上追踪一条完整的信号路径——从点击按钮到舵机转动，中间经过多少环节。

**学到的方法**：
- 分层之后一定要做**端到端追踪**，否则分层只是纸上谈兵
- 追踪信号时关注三个指标：**延迟、带宽、可靠性**
- 每个环节问自己：这里会不会丢数据？会不会断？断了怎么办？

### 第 3 问：「这些知识点应该怎么组织固化？」

```
Q: "这算是学习经验与路线的一部分吧，我需要贯彻"总结一套快速高效学习方法"的路线，
    请你把这些内容分为通讯层，飞机层的知识，写入md文件"
```

**这一问的价值**：从理解走向沉淀。学完不等于掌握，写下来才等于掌握。你在主动把零散的问答整理成结构化的知识文档。

**学到的方法**：
- **输出倒逼输入**：试图写清楚一个概念时，才发现自己哪里没真懂
- 学习笔记不是抄书，而是**用自己的语言重新组织**一遍
- 好的笔记有两个标准：① 能给后来的队友当教材 ② 一个月后自己还能看懂

## 三、通讯方式总结

### 通讯层涉及的五种通讯协议

> 记法：**"由近到远，从线到波"**——从飞控旁的串口线开始，到 4G 电磁波结束。

```
距离:  10cm          30cm           数公里          数百公里
       ┌───┐      ┌───────┐      ┌───────┐      ┌──────────┐
       │UART│ ──→ │  USB  │ ──→  │4G LTE │ ──→  │   UDP    │
       │串口│      │ 总线   │      │ 空口   │      │  用户数据报│
       └───┘      └───────┘      └───────┘      └──────────┘
       MAVLink      MAVLink       IP包里的        MAVLink
       原始字节      原始字节       MAVLink字节      原始字节
```

### 协议一：UART 串口通信（飞控 ↔ 树莓派）

**物理层**：
- 两根线：TX（发送）、RX（接收）+ GND（共地参考）
- 3.3V 电平，交叉连接（TX→RX, RX→TX）
- DM-FC01 TELEM1 (USART2) ↔ 树莓派 GPIO14/15

**协议层**：
- 波特率 115200（每秒 115200 个 bit，约 11.5 KB/s）
- 数据帧：1 起始位 + 8 数据位 + 1 停止位 + 无校验位（8N1）
- 全双工：收发可以同时进行

**学习方法**：
1. **先做回环测试**：短接 TX 和 RX，自己发自己收，确认串口硬件正常
2. **再用示波器看波形**：理解什么叫"起始位"、"停止位"，比看十遍教材管用
3. **最后抓 MAVLink 数据**：用 `hexdump` 或串口监视器看真实的 MAVLink 字节流
4. 关键参数只记两个：**波特率**（影响速度）和**电平**（接错会烧）

### 协议二：USB 总线（树莓派 ↔ EC20）

**物理层**：
- 4 线：VBUS(5V), D+, D-, GND
- EC20 作为一个 USB Device，树莓派是 USB Host
- 速率：USB 2.0 High-Speed (480 Mbps，足够 EC20 用)

**协议层**：
- EC20 向树莓派呈现**多个 USB 接口**：
  - `/dev/ttyUSB0` — DM 口（诊断/调试）
  - `/dev/ttyUSB1` — NMEA 口（GPS NMEA 输出）
  - `/dev/ttyUSB2` — AT 命令口（配置模块用）
  - `/dev/ttyUSB3` — PPP/QMI 数据口
  - `wwan0` — 拨号成功后出现的网卡接口
- 树莓派通过 AT 命令配置 EC20，通过 QMI/PPP 协议建立数据连接

**学习方法**：
1. **不要一上来就读 AT 命令手册**（几百条命令，吓死人）
2. 先掌握 **5 条核心 AT 命令**：`AT`（活着吗）、`AT+CSQ`（信号）、`AT+COPS?`（运营商）、`AT+QCFG="usbnet"`（网卡模式）、`AT+CPIN?`（SIM 卡状态）
3. 遇到新需求再去手册里查
4. EC20 最难的永远是**供电**和**天线**，不是 AT 命令

### 协议三：4G LTE 空口（EC20 ↔ 基站）

**物理层**：
- EC20 支持的 LTE 频段：B1(2100), B3(1800), B5(850), B8(900), B38(TD2600), B39(TD1900), B40(TD2300), B41(TD2500)
- 中国三大运营商常用频段：移动 B3/B8/B39/B40/B41，联通 B1/B3/B8，电信 B1/B3/B5
- 天线接口：IPEX 一代座子（两种：主天线 + 分集天线）

**关键特性**：
- 上行速率 5-10 Mbps（决定视频推流质量上限）
- 下行速率 20-50 Mbps
- RTT 延迟 30-80ms（空口 10-20ms + 核心网路由 20-60ms）
- 飞机高度越高，信号越差（基站天线向下覆盖，高空不在主瓣内）

**学习方法**：
1. 4G 这一层**你不需要像学串口那样深究物理层**——你只需要知道怎么判断信号好坏
2. 核心技能：`AT+CSQ` 读信号强度，`AT+QENG="servingcell"` 看小区信息
3. 飞行中记录信号强度 vs 高度 vs 距离的关系，做出自己的信号覆盖热力图
4. 关键认知：**4G 比你想象的不可靠**——丢包、断连、切换基站，都是常态。你的系统要能容忍这些

### 协议四：UDP/IP（树莓派 ↔ 服务器）

**为什么用 UDP 而不是 TCP？**

| | UDP | TCP |
|------|------|------|
| 可靠性 | 不保证送达 | 保证送达（重传） |
| 延迟 | 低（无握手） | 高（三次握手+ACK） |
| 适用场景 | **实时遥测、控制指令** | 文件上传、日志同步 |
| MAVLink 配合 | ✅ 天然匹配（MAVLink 自己有心跳+超时） | ❌ TCP 重传会导致 MAVLink 消息乱序 |

**特征**：
- 树莓派主动向 `8.152.207.82:14550` 发第一个 UDP 包，打洞 NAT
- 服务器收到后记录源 IP+端口，之后可以随时回复
- MAVLink 的心跳机制保证链路存活检测（5 秒无心跳 → 判定断连）

**学习方法**：
1. 先用 `nc`（netcat）做最简单的 UDP 收发测试，不涉及 MAVLink
2. 理解 NAT 的本质：内网能主动连外网，外网不能主动连内网
3. 抓包！`tcpdump -i wwan0 port 14550` 看真实的 UDP 包大小和间隔

### 协议五：MQTT（服务器内部，可选桥接到飞机）

**在通讯层的角色**：非实时消息（状态上报、日志、配置下发），不替代 MAVLink。

**学习方法**：MQTT 协议极简（固定头部 2 字节），一晚上读完。重点在实际搭建——在服务器上搭 Mosquitto，用 `mosquitto_sub` 验证。

### 通讯协议学习优先级

```
优先级 1 (必须先掌握):
  UART 串口  —— 飞控和树莓派通信的基础，不懂就无法调试
  UDP/IP    —— 飞机能连上服务器的关键

优先级 2 (尽快掌握):
  AT 命令    —— 配置 EC20 模块的必备技能
  MAVLink    —— 不用懂消息格式，但要懂"透明转发"的概念

优先级 3 (后期深入):
  MQTT      —— 服务器端消息总线
  4G 空口特性 —— 信号优化、频段选择
```

## 四、通讯层学习方法论

### 方法一：从 "物理上怎么接" 开始

> 不要一上来就读协议文档。先拿实物，把线接上，灯亮了再说。

通讯层的学习最怕的是**在脑子里跑数据**——以为理解了 UART，但实际连 TX 接 RX 都不知道。

**正确路径**：
```
第 1 步：拿出飞控和树莓派，用杜邦线把 TELEM1 和 GPIO 连好
第 2 步：树莓派打开串口，飞控上电，看有没有字节流
第 3 步：有字节流了 → 再去读 MAVLink 协议，理解这些字节是什么
第 4 步：理解了 → 插上 EC20，AT 命令拨号，ping 通外网
第 5 步：mavproxy 配通 → 服务器收到数据 → 完成！
```

**反例（不要这样学）**：先把 MAVLink 协议文档 200 页读完 → 再读 EC20 AT 命令手册 500 页 → 再读 RFC 768 (UDP) → 最后实际接线，发现串口线接反了。

### 方法二：每一段都做 "回环测试"

通讯调试最大的噩梦是：链路不通，但不知道断在哪一段。回环测试 = 把每一段闭环起来独立验证。

```
┌─────────────────────────────────────────────────────────────────┐
│  分段回环测试方案                                                 │
│                                                                 │
│  段① 飞控↔树莓派:                                                │
│    树莓派短接 GPIO TX+RX → 自发自收 → 串口硬件 OK                 │
│    飞控 USB 连电脑 → QGC 收到数据 → 飞控 MAVLink 输出 OK          │
│    两端都 OK → 连在一起 → 必然 OK                                │
│                                                                 │
│  段② 树莓派↔EC20:                                                │
│    EC20 AT 口发 AT → 回 OK → EC20 活着                           │
│    AT+CSQ → 回信号值 → SIM 卡 + 天线 OK                          │
│    ping 8.8.8.8 → 通 → 互联网 OK                                │
│                                                                 │
│  段③ 互联网↔服务器:                                              │
│    从任意能上网的电脑 UDP 发包到 8.152.207.82:14550              │
│    服务器 tcpdump 看到包 → 服务器端口 OK                         │
│                                                                 │
│  段④ 服务器↔地面站(QGC):                                        │
│    mavlink-router 转发到 14551 → QGC 连上 → 看得到飞机           │
└─────────────────────────────────────────────────────────────────┘
```

### 方法三：学会抓包和看原始数据

通讯调试最有力的工具不是文档，是 **tcpdump** 和 **串口监视器**。

```bash
# 飞机端 (树莓派): 看串口有没有数据
cat /dev/serial0 | hexdump -C

# 飞机端 (树莓派): 看发出了什么 UDP 包
tcpdump -i wwan0 -X port 14550

# 服务器端: 看收到了什么 UDP 包
tcpdump -i eth0 -X port 14550
```

**学到什么程度算掌握？**：不看 MAVLink 文档，看 hexdump 的原始字节，能认出 `FD`（MAVLink v2 起始标志）和心跳包的固定模式（每 1 秒一条，长度固定的包）。

### 方法四：建立"延迟预算"概念

通讯层的核心设计约束不是功能，而是**延迟**。

```
延迟预算 (从服务器下指令到飞控执行):

  服务器内部:       < 1ms   (pymavlink → mavlink-router → UDP 发包)
  公网路由:        5-15ms   (BGP 路由，光纤)
  运营商核心网:    10-30ms   (NAT + 路由)
  4G 空口:         10-40ms   (LTE RTT，取决于信号)
  树莓派内部:       < 1ms    (UDP 收 → 串口发)
  串口传输:         < 1ms    (115200, 一帧 MAVLink ~20-200 字节)
  飞控解析+执行:    < 1ms    (中断→协议栈→PWM 输出)
  ─────────────────────────
  总计:            30-80ms
```

这 30-80ms 对于投放包裹、切换模式、返航等操作完全够用。但如果是**视觉伺服**（ArUco 定位调整姿态），80ms 就太慢了——这就是为什么 ArUco 必须跑在飞机端（树莓派本地处理 → 直接发 MAVLink，不走 4G）。

## 五、通讯层同学的自检清单

按顺序逐一通过，前面的不通不要往后走：

- [ ] **串口通了**：飞控 TELEM1 接树莓派 GPIO，`cat /dev/serial0` 有字节输出
- [ ] **QGC 直连**：飞控 USB 连 QGC，能看姿态 + 能设参数
- [ ] **串口回环**：树莓派短接 TX+RX，Python 自发自收正确
- [ ] **AT 命令**：EC20 插树莓派，`AT` 回 `OK`，`AT+CSQ` 信号 ≥ 15
- [ ] **4G 上网**：EC20 拨号成功，`ping 8.8.8.8` 通
- [ ] **UDP 通**：树莓派 UDP 发到服务器 `8.152.207.82:14550`，服务器 tcpdump 看到
- [ ] **MAVLink 端到端**：飞控数据经过树莓派→EC20 到达服务器，QGC 连上服务器能看到飞机
- [ ] **下行指令**：从 QGC 发指令，飞控能收到并执行（比如切到 Guided 模式）
- [ ] **断连恢复**：拔掉 EC20 天线 → 确认飞控触发 RTL failsafe → 插回天线 → 链路恢复

AT+CSQ                                  # 信号强度 (0-31, ≥15 可用)
AT+COPS?                                # 在哪个运营商
AT+CPIN?                                # SIM 卡状态 (READY=正常)
AT+QCFG="usbnet",2                      # 设网卡模式为 QMI

# 网络相关
ifconfig wwan0                          # 看 4G 网卡有没有 IP
ping -I wwan0 8.8.8.8                  # 通过 4G 网卡 ping
route -n                                # 默认路由走了 4G 吗？

# mavproxy
mavproxy.py --master=/dev/serial0 --out=8.152.207.82:14550
pkill -9 -f mavproxy                   # 杀干净

# 抓包
tcpdump -i wwan0 -n port 14550         # 看 UDP 包
tcpdump -i wwan0 -n port 14550 -X      # 看包内容（hex+ASCII）

# === 服务器端 ===

# mavlink-router
mavlink-routerd -e 127.0.0.1:14551 0.0.0.0:14550

# 监控
tcpdump -i eth0 -n port 14550          # 看收到了什么
ss -tulnp | grep -E '14550|1883'       # 端口在监听吗

# MQTT
mosquitto_sub -v -t 'drone/#'          # 订阅所有飞机消息
systemctl status mosquitto              # Broker 状态
