# 飞机层 — 学习笔记

> **所属层级**：四层架构之飞机层
> **负责同学**：馨淼、哲宇（飞控+ArduPilot）、秋森、用户（3D建模+结构）
> **核心职责**：飞行姿态稳定、MAVLink 指令执行、ArUco 视觉定位、动力驱动。**所有延迟 < 50ms 的操作必须在飞机层完成。**

### MAVLink 协议的四层子结构

MAVLink 不只是"一个协议"——它内部有四层子结构，从底层的二进制帧格式到上层的任务级交互。理解这四层，才能理解**飞控和外界的每一次对话到底在发生什么**。

```
┌─────────────────────────────────────────────────────────────────┐
│                 MAVLink 协议栈 四层子结构                         │
│                                                                 │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  第 4 层: 交互协议层 (Protocol Mechanisms)                  │  │
│  │                                                              │  │
│  │  心跳机制:    飞控每秒发 HEARTBEAT → 对方 5 秒收不到就断连  │  │
│  │  指令确认:    发 COMMAND_LONG → 飞控回 COMMAND_ACK (接受/拒绝)│  │
│  │                 → 这是你判断"指令发成功了没有"的唯一依据      │  │
│  │  参数协议:    读 PARAM_REQUEST_READ → 回 PARAM_VALUE         │  │
│  │              写 PARAM_SET → 回 PARAM_ACK (成功/失败)         │  │
│  │  航线协议:    发 MISSION_COUNT → MISSION_REQUEST_ITEM 逐条    │  │
│  │               上传航点 → MISSION_ACK 确认                     │  │
│  │  消息间隔:    请求某条消息的发送频率 (REQUEST_DATA_STREAM)    │  │
│  │                                                              │  │
│  │  ⚠️ 很多初学者只说"MAVLink 命令发过去了"，但从不检查 ACK     │  │
│  │     没有 ACK = 你不知道飞控是否真的收到了指令                 │  │
│  └───────────────────────────────────────────────────────────┘  │
│                            ↑                                     │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  第 3 层: 消息类型层 (Message Types / Dialects)              │  │
│  │                                                              │  │
│  │  按功能分类 (ArduPilot 常用):                                 │  │
│  │                                                              │  │
│  │  HEARTBEAT (#0)        心跳+系统类型+飞行模式                │  │
│  │  SYS_STATUS (#1)       电池电压/电流/剩余                    │  │
│  │  GPS_RAW_INT (#24)     GPS 经纬度/高度/速度                  │  │
│  │  ATTITUDE (#30)        姿态角 roll/pitch/yaw                 │  │
│  │  RC_CHANNELS (#65)     遥控器各通道值                         │  │
│  │  COMMAND_LONG (#76)    发送指令 (最常用)                      │  │
│  │  COMMAND_ACK (#77)     指令回执 (接受/拒绝/失败)             │  │
│  │  PARAM_REQUEST_READ (#20)  读参数                            │  │
│  │  PARAM_VALUE (#22)         参数值返回                        │  │
│  │  MISSION_ITEM (#39)        单个航点数据                       │  │
│  │  STATUSTEXT (#253)     飞控文本状态 (info/warning/error)     │  │
│  │                                                              │  │
│  │  Dialect: ArduPilot 使用 ardupilotmega.xml 定义的消息集      │  │
│  │  每条消息有唯一 ID (0-255)，payload 结构在 XML 里定义         │  │
│  └───────────────────────────────────────────────────────────┘  │
│                            ↑                                     │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  第 2 层: 帧格式层 (Frame Format)                            │  │
│  │                                                              │  │
│  │  MAVLink v2 帧结构 (4-280 字节):                             │  │
│  │                                                              │  │
│  │  ┌──────┬────┬────┬────┬────┬────┬───────────┬──────┬────┐  │  │
│  │  │ STX  │LEN │SEQ │SYS │COMP│MSG │ PAYLOAD   │CKSUM │SIG │  │  │
│  │  │ 0xFD │1B  │1B  │1B  │1B  │3B  │ 0-255 字节 │ 2B   │13B │  │  │
│  │  └──────┴────┴────┴────┴────┴────┴───────────┴──────┴────┘  │  │
│  │   固定    │         │         │      │           可选签名     │  │
│  │   魔术字   │   6 字节头部       │      │    (v2 独有)          │  │
│  │           │                    │      │                      │  │
│  │  STX:   起始标志 0xFD (v2) / 0xFE (v1)                      │  │
│  │  LEN:   payload 长度 (0-255)                                 │  │
│  │  SEQ:   包序号 (0-255 循环, 用于检测丢包)                    │  │
│  │  SYS:   系统 ID (飞控=1, 地面站=255)                         │  │
│  │  COMP:  组件 ID (飞控本体=1, 云台=100, 相机=200...)          │  │
│  │  MSG:   消息 ID, 3 字节 (最大 16M 种, 这决定了消息类型)     │  │
│  │  CKSUM: CRC-16/MCRF4XX 校验 (收方算一遍, 不匹配→丢包)       │  │
│  │  SIG:   签名 (防伪造, 需配置密钥才生效)                       │  │
│  │                                                              │  │
│  │  ⚠️ 串口上看到的不是文本, 是二进制。0xFD 开头的一串字节       │  │
│  │  hexdump 里认得出 MAVLink 包: 总是以 FD 开头                  │  │
│  └───────────────────────────────────────────────────────────┘  │
│                            ↑                                     │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  第 1 层: 传输适配层 (Transport Layer / Channels)            │  │
│  │                                                              │  │
│  │  MAVLink 不关心底层传输介质。同一套消息可以跑在:              │  │
│  │                                                              │  │
│  │  Serial (UART)     ← 飞控 TELEM1 ↔ 树莓派 (本项目主链路)    │  │
│  │  UDP               ← 树莓派 ↔ 云服务器                       │  │
│  │  TCP               ← QGC 地面站 ↔ mavlink-router             │  │
│  │  USB CDC ACM       ← 飞控 USB ↔ 电脑 (调试用)               │  │
│  │  CAN               ← 飞控 ↔ CAN 设备                         │  │
│  │                                                              │  │
│  │  传输层只需要提供: 收发字节流的能力                           │  │
│  │  → MAVLink 帧就是一串字节, 任何能传字节的信道都能用          │  │
│  │                                                              │  │
│  │  实际收发流程:                                                │  │
│  │                                                              │  │
│  │  发送端:                                                      │  │
│  │    构造消息结构体 (C struct)                                 │  │
│  │    → mavlink_msg_xxx_encode()  序列化为字节数组              │  │
│  │    → mavlink_msg_to_send_buffer()  封帧 (加 STX+头部+CRC)    │  │
│  │    → UART 发送 / UDP sendto()                                │  │
│  │                                                              │  │
│  │  接收端:                                                      │  │
│  │    UART 收到字节 → 逐字节喂入 mavlink_parse_char()           │  │
│  │    → 状态机: 等 STX → 收头部 → 收 payload → 核对 CRC         │  │
│  │    → 拼完整 → 调 mavlink_msg_xxx_decode() 解析为结构体       │  │
│  │    → 根据 msgid 派发到对应 handler 函数                       │  │
│  │                                                              │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  ═══════════════════════════════════════════════════════════    │
│                                                                 │
│  四层对照记忆法:                                                 │
│                                                                 │
│  寄快递  ┆  MAVLink                                             │
│  ────────┼──────────────────────────────────────                 │
│  快递流程  ┆  第4层 交互协议  (发件→确认→签收)                  │
│  货物种类  ┆  第3层 消息类型  (文件/包裹/冷链, 各有格式)        │
│  打包方式  ┆  第2层 帧格式    (箱子+填充+胶带+标签)             │
│  运输公司  ┆  第1层 传输适配  (顺丰/圆通/EMS, 都能送货)         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> **飞机层同学必读**：第 2 层（帧格式）和第 4 层（交互协议）是理解飞控通信的核心。你能从 hexdump 里认出 `FD` 开头，你就不会再被"串口乱码"吓到。你能理解 COMMAND_ACK，你就知道指令到底有没有被飞控接受。

## 二、提问路线回顾

> 以下是你在学习飞机层过程中自然形成的提问路径。这条路径本身就是一套高效的学习方法论——**从分层定位到端到端追踪，从协议理解到物理执行**。

### 第 1 问：「项目怎么分成不同层？」

```
Q: "我可以不可以把整个项目按分工分为飞机层，电站层，通讯层，服务层"
```

**这一问的价值**：把飞机层从整个系统中**剥离**出来，明确它的边界和职责。飞机层的核心问题变成了：**"我和谁通信？我需要输出什么信号？我需要保证什么性能？"**

**学到的方法**：
- 飞机层的边界 = TELEM1 串口（向北，对通讯层）+ ArUco 码/触点（向下，对电站层）
- 凡是延迟 < 50ms 的操作，归飞机层（姿态、视觉定位、failsafe）
- 凡是延迟 > 100ms 的操作，往外推（调度、存储、界面）

### 第 2 问：「MAVLink 是怎样实际给飞控发指令的？」

```
Q: "给我飞机->树莓派->网卡->服务器的通信链路，以及mavlink是怎样实际给飞控发指令的"
```

**这一问的价值**：这是飞机层最核心的问题——**飞控怎么听懂并执行来自外部的指令？**

你追踪了一条完整的指令路径：UDP 包 → 树莓派 → 串口字节流 → USART2 中断 → 协议栈逐字节解析 → 消息 ID 查表 → handler 函数 → PWM 输出 → 舵机旋转。这条路径上每一环都问了"为什么"。

**学到的方法**：
- **端到端追踪一条信号**是理解嵌入式系统最快的方法
- 从用户操作（点"投放"按钮）回溯到物理信号（PB1 引脚上的 PWM 波形），一个环节不漏
- 每个环节标注延迟，建立"延迟预算"意识

### 第 3 问：「这些知识应该怎么按层组织？」

```
Q: "这算是学习经验与路线的一部分吧，我需要贯彻"总结一套快速高效学习方法"的路线，
    请你把这些内容分为通讯层，飞机层的知识，写入md文件"
```

**这一问的价值**：从理解走向体系化。你意识到飞机层的知识不是孤立的——它和通讯层、电站层有明确的接口，需要通过分层文档把边界固化下来。

**学到的方法**：
- 分层文档 = **接口契约**：写清楚层与层之间的物理连接、协议、约定
- 好的分层文档有两个读者：① 本层同学（深入理解内部原理）② 邻层同学（只需要知道接口）

## 三、飞机控制方式总结

### 控制方式全景图

飞机层的控制分为两大范畴：**飞控自己能做的事**（不需外部指令）和**需要外部指令的事**（通过 MAVLink 触发）。

```
┌─────────────────────────────────────────────────────────────────┐
│                    飞机层 控制方式分类                           │
│                                                                 │
│  ┌─────────────────────────┐    ┌─────────────────────────┐    │

│  │                         │    │  航点上传  MISSION_*    │    │
│  └─────────────────────────┘    └─────────────────────────┘    │
│                                                                 │
```

### 控制方式一：MAVLink 指令 → PWM 输出（最直接）

这是你们项目最常用的控制路径——投放包裹、弹降落伞。

```
外部 (服务器/QGC)
      │
      │  MAV_CMD_DO_SET_SERVO
      │  param1 = 5 (SERVO5), param2 = 2000 (PWM 值)
      ▼
飞控 MAVLink 协议栈
      │
      │  消息 ID = 183 → GCS_Common.cpp handle_command_long()
      ▼
SRV_Channels::set_output_pwm(5, 2000)
      │
      │  写 TIM3 比较寄存器
      ▼
STM32 TIM3_CH4 → PB1 引脚 → 2000μs 脉宽 PWM
      │
      ▼
投放舱舵机 → 旋转到指定角度 → 舱门打开
```

**关键参数**：
- PWM 频率：50Hz（20ms 周期），标准舵机协议
- PWM 范围：1000μs（最小）～ 2000μs（最大），1500μs 为中心
- 对应舵机角度：1000→0°, 1500→90°, 2000→180°（典型值，实际需校准）

**为什么用 PWM 而不是串口/I²C/SPI？**
- PWM 是飞控最原生、最可靠的输出方式
- 不改固件，不定义新 MAVLink 消息
- 只要模块支持 PWM 输入，就能被飞控控制
- 飞控不关心 AUX 引脚上接的是什么——舵机、继电器、电热丝，都一样

### 控制方式二：飞行模式切换（最灵活）

ArduPilot 有多种飞行模式，每种模式是一套完整的控制逻辑：

| 模式 | MAVLink 指令 | 飞控做什么 | 适用场景 |
|------|-------------|-----------|---------|
| **Stabilize** | MODE=0 | 自稳（手动飞） | 起飞、调试 |
| **Loiter** | MODE=5 | GPS 悬停 | 等待指令 |
| **Guided** | MODE=4 | 接受外部位置指令 | 服务器控制飞行 |
| **Auto** | MODE=3 | 执行预设航线 | 全自动投递 |
| **RTL** | MODE=6 | 返航+降落 | 任务完成/异常 |
| **Land** | MODE=9 | 原地降落 | 精确降落（配合 ArUco） |

**如何切换**：
```
# 服务器端 (pymavlink)
# 切换到 Guided 模式
conn.mav.command_long_send(
    target_system, target_component,
    mavutil.mavlink.MAV_CMD_DO_SET_MODE,
    0,
    1,    # base_mode: custom
    4,    # custom_mode: Guided (ArduPilot 编号)
    0, 0, 0, 0, 0
)
```

### 控制方式三：飞控自主 Failsafe（最可靠——不需要通讯层）

这是飞机层的**安全底线**。以下所有保护机制都在飞控固件里，**设参数即可启用，不需要写代码**：

| Failsafe | 触发条件 | 飞控动作 | 关键参数 |
|----------|---------|---------|---------|
| **低电量** | 电压 < `FS_BATT_VOLTAGE` | RTL 返航 | `FS_BATT_ENABLE=1`, `FS_BATT_VOLTAGE=14.0` (4S) |
| **4G 断连** | 5 秒无 MAVLink 心跳 | RTL 返航 | `FS_GCS_ENABLE=1`, `FS_LOST_TIMEOUT=5` |
| **遥控断连** | 接收机无信号 | RTL 返航 | `FS_THR_ENABLE=1` |
| **GPS 丢星** | HDOP 过大或卫星数不足 | 切 Stabilize + 降落 | `FS_GPS_ENABLE=1` |
| **电子围栏** | 飞出设定范围 | 立即 RTL | `FENCE_ENABLE=1`, `FENCE_RADIUS=500` (米) |
| **降落伞自触发** | 姿态异常（翻覆/坠落） | 弹降落伞 | `CHUTE_ENABLED=1` |

**核心认知**：通讯层可以断，服务器可以崩，但飞控必须能在无人干预的情况下自己安全回家。这是飞机层最根本的设计原则。

### 控制方式四：ArUco 视觉定位（树莓派本地闭环——不走 4G）

```
[摄像头] ──→ [树莓派 3B] ──→ [飞控 MAVLink]
               │                 │
               │ OpenCV          │ 收到位置修正指令
               │ 检测 ArUco 码    │ 调整姿态 → 精确降落
               │ 计算偏移量       │
               │                 │
               │<── 整个闭环 ──→│
               │   延迟 <50ms     │
               │   不走 4G        │
```

**为什么必须在飞机层闭环？**
- 4G 链路延迟 30-80ms，加上图像上传时间，总延迟 > 200ms
- 降落过程中 200ms 的延迟意味着 2m/s 下降时位置偏差 40cm——ArUco 码可能已经飞出视野
- 树莓派本地处理 → 直接通过 UART MAVLink 发位置修正指令给飞控 → 延迟 < 50ms

## 四、飞机层学习方法论

### 方法一：从"外设驱动"学起，不是从"源码"学起

> ArduPilot 代码有 100 万行以上。试图"通读源码"是最高效的自杀方式。

**正确路径**：
```
第 1 步：飞控连 QGC，看参数树，理解每个参数分类
第 2 步：飞控连 Mission Planner，用 MAVLink Inspector 看实时消息流
第 3 步：手动拿飞控动一动，看 IMU 数据怎么变
第 4 步：找到你想改的功能对应的参数，先调参，不改代码
第 5 步：如果参数不够用，再去找到对应的源码文件，只读那个文件
第 6 步：改一小段代码，编译烧录，看效果
```

**ArduPilot 源码导航（只读你需要的那部分）**：

| 你想做什么 | 去哪看 | 关键文件 |
|-----------|--------|---------|
| 理解 MAVLink 指令怎么处理 | `libraries/GCS_MAVLink/` | `GCS_Common.cpp` |
| 加一个自定义飞行模式 | `ArduCopter/` | `mode.h`, `mode.cpp` |
| 控制舵机/PWM 输出 | `libraries/SRV_Channel/` | `SRV_Channel.cpp` |
| 理解姿态控制 PID | `libraries/AC_AttitudeControl/` | `AC_AttitudeControl.cpp` |
| 加一个新传感器驱动 | `libraries/AP_InertialSensor/` | 看现有传感器的写法 |
| 改 LED/蜂鸣器行为 | `libraries/Notify/` | `AP_Notify.cpp` |
| 理解 failsafe 逻辑 | `ArduCopter/` | `events.cpp` |

### 方法二：用 MAVLink Inspector 建立直觉

QGC / Mission Planner 都有 **MAVLink Inspector** 功能——显示飞控发出的每一条 MAVLink 消息的原始字段。

**实操训练**：
1. 飞控 USB 连电脑，打开 MAVLink Inspector
2. 找到 `HEARTBEAT` 消息——看它的 `type` 字段（2=多旋翼）、`autopilot` 字段（3=ArduPilot）
3. 倾斜飞控——看 `ATTITUDE.roll` 数值变化，建立"物理动作 ↔ 数值"的直觉
4. 找到 `GPS_RAW_INT`——看 `lat/lon` 是原始值（需除以 1e7）、`alt` 的变化
5. 找到 `SYS_STATUS`——看 `voltage_battery`（需除以 1000）、`current_battery`
6. 用遥控器打杆——看 `RC_CHANNELS` 消息各通道的值（1000-2000）

**学到什么程度算掌握？**：不看 MAVLink 文档，看 Inspector 里的消息名和字段，能猜出每条消息大概干什么用。

### 方法三：建立"安全底线思维"

飞机层和通讯层/服务层最大的区别是：**飞机层的 bug 会炸机**。

```
飞机层 安全设计原则:

1. Failsafe 永远是最后一道防线
   → 通讯断了？飞控自己 RTL
   → 低电压？飞控自己 RTL
   → GPS 丢了？飞控自己切 Stabilize 降落
   → 姿态异常？飞控自己弹降落伞
   所有这些都必须在飞控固件里完成，不依赖任何外部设备

2. 任何外部指令都不能绕过 failsafe
   → 服务器发"投放"指令，但电池电压已经很低？
   → 飞控应该拒绝执行，先 RTL

3. 地面验证 30 分钟以上才能上天
   → 改一行代码 → 地面跑 30 分钟 → 确认不影响核心飞控功能
   → 新功能第一次上机测试 → 低空（2m）+ 系留绳

4. 每一路 PWM 输出要有独立供电
   → 投放舱舵机从独立 BEC 取电，不要从飞控 5V 取
   → 舵机堵转电流 1-2A，飞控 5V BEC 只有 3A 总输出
```

### 方法四：参数驱动开发

ArduPilot 有 500+ 个参数。你们项目中 80% 的需求，调参就能解决。

**项目关键参数速查**：

| 参数分类 | 关键参数 | 作用 |
|----------|---------|------|
| **串口配置** | `SERIAL1_PROTOCOL=2`, `SERIAL1_BAUD=115` | TELEM1 = MAVLink2 给树莓派 |
| **电池监测** | `BATT_MONITOR=4`, `BATT_VOLT_MULT=10.2` | 电压分压校准 |
| **Failsafe** | `FS_GCS_ENABLE=1`, `FS_LOST_TIMEOUT=5` | 5 秒断连 → RTL |
| **降落伞** | `CHUTE_ENABLED=1`, `CHUTE_SERVO_ON=6` | SERVO6 触发降落伞 |
| **电子围栏** | `FENCE_ENABLE=1`, `FENCE_RADIUS=500` | 超 500m 自动返航 |
| **舵机输出** | `SERVO5_FUNCTION=0`, `SERVO5_MIN=1000`, `SERVO5_MAX=2000` | AUX1 作通用输出 |
| **RTL 高度** | `RTL_ALT=3000` (30m) | 返航前先爬升到安全高度 |

## 五、飞机层同学的自检清单

按顺序逐一通过：

### 基础操作

- [ ] **飞控点亮**：USB 接电脑，QGC 连上，确认飞控型号和固件版本
- [ ] **看数据流**：MAVLink Inspector 找到 HEARTBEAT、ATTITUDE、GPS_RAW_INT、SYS_STATUS
- [ ] **校准 IMU + 罗盘**：QGC 校准向导走完，确认姿态显示准确
- [ ] **刷固件**：PX4 ↔ ArduPilot 来回刷 3 遍以上，不怕变砖

### 接口验证

- [ ] **串口输出验证**：飞控 TELEM1 接 USB-TTL，电脑串口助手看 MAVLink 字节流
- [ ] **PWM 输出验证**：示波器测 PWM1-4（电机口）有 DShot 信号，PWM5-8 有 50Hz PWM
- [ ] **接收机验证**：ELRS 接收机接 UART5，QGC 看 RC_CHANNELS 有值

### 控制验证

- [ ] **MAVLink 指令**：从 QGC 发 `MAV_CMD_DO_SET_SERVO`，示波器看 SERVO5 输出变化
- [ ] **模式切换**：QGC 切换 Stabilize → Loiter → Guided → RTL，确认飞控响应
- [ ] **Failsafe**：拔掉接收机 → 确认飞控触发 FS_THR（需要设参数后测试）

### 联调准备

- [ ] **ArUco 地面测试**：树莓派 + 摄像头对着 ArUco 码，能输出偏移量
- [ ] **投放舱地面测试**：从 QGC 发 SERVO5=2000，舵机正确转动，舱门正常开启
- [ ] **降落伞地面测试**：从 QGC 发 DO_PARACHUTE，SERVO6 输出正确 PWM
